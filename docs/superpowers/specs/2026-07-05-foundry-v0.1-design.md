# Foundry v0.1 — Design Spec

Date: 2026-07-05
Status: approved by Mario (brainstorming session, 2026-07-05)

## What Foundry is

Foundry is a design system library for Android built with Jetpack Compose: design tokens,
themed components and a consistent visual language, ready to drop into any app. It is the
second portfolio project after Conduit and follows the same engineering discipline: design
units delivered one PR at a time, squash-merged with Conventional Commits, published to
Maven Central under `io.github.marioponceg`.

**Relationship with Material 3 (settled):** Foundry is an own layer on top of M3. It defines
its own token system and theme; under the hood `FoundryTheme` installs a mapped
`MaterialTheme` so wrapped M3 components inherit the styling. Components wrap M3
counterparts behind Foundry's own API. Foundry does not reimplement components from
foundation, and it is not a mere M3 theming helper.

## Architecture and modules

```
Foundry/
├── build-logic/           # convention plugin foundry.library
├── foundry-tokens/        # com.android.library, published
├── foundry-components/    # com.android.library, published, api(foundry-tokens)
└── catalog/               # com.android.application, NOT published (renamed from app/)
```

### foundry-tokens

The pure design system:

- `FoundryColors` — light and dark color schemes, both shipped in v0.1. Dynamic color
  (Material You) is explicitly out of scope for v0.1; it may come later as opt-in.
- `FoundryTypography` — the type scale.
- `FoundrySpacing` — the spacing scale.
- `FoundryShapes` — the shape scale.
- `FoundryTheme {}` — exposes the tokens via `CompositionLocal`s and installs a mapped
  `MaterialTheme` underneath. Idiomatic access mirrors M3:
  `FoundryTheme.colors.accent`, `FoundryTheme.spacing.md`, etc.
  `darkTheme` parameter defaults to `isSystemInDarkTheme()`.

Dependencies: `compose.runtime`, `compose.ui`, `material3` (theme mapping only — no M3
components are used in this module).

### foundry-components

v0.1 components, each wrapping the M3 counterpart behind Foundry's API with typed variants
(enums/sealed types), slots, and defaults resolved from tokens:

- `FoundryText` — text styles from `FoundryTypography`.
- `FoundryButton` — action variants.
- `FoundryCard` — container with slots.
- `FoundryTextField` — controlled-state input, the most complex API of the set.

Exposes `foundry-tokens` as an `api` dependency (same criterion as
`conduit-engine-okhttp` → `conduit-core`).

### catalog

The showcase app, reoriented from the template's `app/` module. One screen per component
plus a tokens screen (palette, type scale, spacing), with a light/dark toggle. Consumes the
library modules as project dependencies. Never published; it is also the daily development
vehicle for the libraries.

### build-logic

Convention plugin `foundry.library` concentrating: AGP library config, Compose, vanniktech
maven-publish, Roborazzi. Reuses Conduit's publishing lessons: publishing plugin loaded
apply-false at root, `VERSION_NAME` in `gradle.properties` as the single source of truth,
signing conditional on `signingInMemoryKey` presence, `publishToMavenCentral --dry-run` to
verify config, `--no-configuration-cache` on the Central upload.

### Toolchain

AGP 9.x (template ships 9.2.1), Kotlin + Compose via BOM in the version catalog, Java 17
target (template's Java 11 is raised in the tooling PR), minSdk 26, targetSdk 36, package
root `io.github.marioponceg.foundry`.

## Testing strategy

- **Screenshot tests with Roborazzi are the central quality metric**: every component
  variant × light/dark, golden images committed to the repo, verified in CI. Chosen over
  Compose Preview Screenshot Testing (younger, less control) and Paparazzi (AGP
  compatibility lag) because it runs on JVM/Robolectric, integrates with GitHub Actions,
  and is the current community standard.
- Unit tests wherever there is logic (theme mapping, token resolution).
- No emulator/instrumented tests in v0.1.
- No Kover coverage threshold: composables measure poorly; Roborazzi's visual coverage is
  the quality bar. This is documented in AGENTS.md.

## PR roadmap to v0.1.0

One design unit per PR, squash-merge, Conventional Commits, `main` ruleset, green CI
required:

1. **PR #1 (`build:`) — Tooling.** Rename `app/` → `catalog/`; add `build-logic` with
   `foundry.library`; Kotlin/Compose via BOM in the version catalog; Java 17; create empty
   `foundry-tokens` and `foundry-components` skeletons (one trivial test each so CI has
   something to run); GitHub Actions CI (build + tests + lint); AGENTS.md with the settled
   decisions; `main` ruleset.
2. **PR #2 (`feat:`) — Tokens + FoundryTheme.** Colors (light/dark), typography, spacing,
   shapes, `FoundryTheme` with M3 mapping. Unit tests for the mapping; Roborazzi enters the
   build here with a visual sample of the theme.
3. **PR #3 (`feat:`) — Catalog base.** Catalog app structure, tokens screen, light/dark
   toggle. From here on, each component PR adds its own catalog screen.
4. **PR #4 (`feat:`) — FoundryText.**
5. **PR #5 (`feat:`) — FoundryButton.**
6. **PR #6 (`feat:`) — FoundryCard.**
7. **PR #7 (`feat:`) — FoundryTextField.** (Component order follows visual dependency —
   the others use Text — with the most complex API last. Each component PR: component +
   Roborazzi tests of all variants in light/dark + catalog screen.)
8. **PR #8 (`build:`/`ci:`) — Publishing.** Maven Central config + tag-triggered release
   workflow (manual release in Central Portal) + README with quick-start. Then tag
   **v0.1.0**.

## Out of scope for v0.1

- Dynamic color / Material You (future opt-in).
- Additional components (Checkbox, Switch, Chip, Dialog… → v0.2+).
- Instrumented/emulator tests.
- KMP or non-Android targets.

## Settled decisions for AGENTS.md

1. Own layer over M3; own tokens mapped to MaterialTheme; components wrap M3.
2. Modules: `foundry-tokens` + `foundry-components` (tokens exposed as `api`) published
   under `io.github.marioponceg`; `catalog` not published; `build-logic` convention plugin.
3. v0.1 scope: tokens + light/dark theme + Text, Button, Card, TextField.
4. Quality: Roborazzi as central metric; unit tests where there is logic; no Kover, no
   emulator in v0.1.
5. Toolchain: AGP 9.x, Compose BOM, Java 17, minSdk 26, targetSdk 36.
6. Flow: one design unit per PR, squash-merge, Conventional Commits, `main` ruleset,
   green CI, ask Mario before any decision not listed here. Publishing via vanniktech +
   Central Portal, manual release, `VERSION_NAME` in `gradle.properties`.
